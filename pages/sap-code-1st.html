ABAP Development

Data Structure

Internal Table과 Work Area를 기준으로 대량 데이터를 처리하고, Field Symbol로 동적 참조와 반복 처리 비용을 줄이는 방식을 학습했습니다.

Report Program

INITIALIZATION, AT SELECTION-SCREEN, START-OF-SELECTION 흐름을 구분하고 선택 조건 검증과 실행 로직을 분리했습니다.

Screen Logic

Selection Screen, PBO, PAI, Module Pool, Tabstrip, Subscreen 구조를 통해 SAP GUI 화면 이벤트 흐름을 정리했습니다.

ALV Control

Field Catalog, Layout, Event Handler, Local Class를 사용해 리스트 조회 화면과 사용자 이벤트를 구성했습니다.

Object · Function

Function Module, Exception, Class/Method, Static/Instance 호출, Inheritance 접근 범위를 실습 단위로 정리했습니다.

External Data Handling

BDC, ADBC, Native SQL 접근을 다루며 SAP 표준 화면 자동 입력과 DB 직접 조회의 역할 차이를 확인했습니다.

CO Process Learning

CO는 표준원가, 실제원가, 배부, 분할, 차이 정산이 연결되는 구조로 정리했습니다. SAP CODE ERP 프로젝트에서는 이 흐름을 기준으로 조직 구조, 원가요소, SKF, 활동단가, Fiori 분석 화면을 설계했습니다.

Standard Cost CK11N · KEKO

표준원가 추정 결과를 월중 원가 비교의 기준값으로 사용

Internal Order AUFK · ACDOCA

임시 비용 집계 단위에서 발생한 비용을 추적하고 정산 대상으로 연결

Settlement KO88 · COBRB

오더 잔액을 정산 규칙에 따라 Cost Center 또는 Profit Center로 이동

SKF COEPR

인원수, 면적, 출하건수, 작업시간을 배부 기준 수량으로 관리

Allocation KSU5 · T811C/L

Cycle/Segment, 송신·수신 규칙, SKF 비율로 공통비를 배부

Profitability ACDOCA · PA Segment

매출, 표준원가, 배부원가를 수익성 분석 단위로 집계

Module Study

SAP CODE 과정 중 별도로 모듈 스터디를 모아 진행하며 FI, SD, MM, PP가 CO로 연결되는 지점을 정리했습니다. CO를 중심에 두되, 매출·출고·구매·생산 데이터가 어떤 테이블과 전표 흐름을 통해 관리회계로 들어오는지 함께 확인했습니다.

CO CoreCost Center Accounting

코스트센터를 기준으로 비용을 계획, 집계, 결산, 리포팅하는 구조를 정리했습니다. 기준정보는 코스트센터, 원가요소, 액티비티, 통계주요지표로 나뉘며 월말에는 배부, 차이계산, 액티비티 가격계산으로 이어집니다.

CO ObjectCost Element · Cost Center

원가요소는 CO Object 귀속이 필요한 수익/비용 계정으로 이해했고, 1차 원가요소와 2차 원가요소의 쓰임을 구분했습니다. 코스트센터는 비용 책임 단위이자 예산, 감가상각, 생산 공정, 공통비 배부, 통계 전기 단위로 활용될 수 있음을 정리했습니다.

Overhead간접비 발생 · 준비 · 배부

FI 전표가 발생하면 CO Object와 함께 CO 데이터가 기록되고, 월말에는 SKF를 기준으로 간접비를 배부합니다. Distribution은 1차 원가요소 성격을 유지하고, Assessment는 여러 비용을 2차 원가요소로 묶어 전달하는 방식으로 구분했습니다.

Product Costing표준원가 · 실제원가 · 차이

CK11N, CK24 흐름으로 표준원가를 계산·표시·방출하고, 생산 실행 이후 실제 투입 원가와의 차이를 계산하는 결산 흐름을 정리했습니다. KEKO, CKIS, MBEW를 표준원가 분석의 핵심 데이터로 연결했습니다.

Material Ledger자재원장과 실제원가

자재원장을 재고 수불 거래의 보조원장으로 보고, 결산 시 원가 차이를 자재 수불에 따라 roll-up하는 실제원가 계산 구조를 학습했습니다. 표준가격으로 운영되는 월중 흐름과 월말 실제원가 보정의 차이를 구분했습니다.

ProfitabilityCO-PA · PCA

CO-PA는 고객, 제품, 시장 관점의 수익성 분석으로, SD 매출과 PP/MM 원가 흐름이 모이는 영역으로 정리했습니다. PCA는 Profit Center 단위 책임회계와 구분회계를 위한 구조로 이해했습니다.

Module Integration Notes

SD → FI/COSales Order · Delivery · Billing

VA01에서 판매 오더가 생성되고, VL01N/VL02N 출고 처리로 재고가 감소하며, VF01 청구 시 FI 전표와 매출 데이터가 생성됩니다. VBRK/VBRP의 청구 데이터는 BKPF/BSEG 회계 전표와 연결되고, VPRS 조건을 통해 표준원가가 CO-PA 분석으로 이어집니다.

PP → CO-PCBOM · Routing · Work Center

PP는 무엇을 만들지, 무엇으로 만들지, 어디서 어떤 순서로 만들지를 정의합니다. Work Center에는 Cost Center와 Activity Type이 연결되어 생산 확정 시 작업시간이 가공비로 전환되고, 표준원가와 실제원가 차이 분석의 기준이 됩니다.

MM → FI/COPurchase Order · Goods Receipt · Invoice

ME21N 구매오더, MIGO 입고, MIRO 송장 검증 흐름에서 계정지정 정보가 CO 연결의 기준이 됩니다. 비용성 구매는 입고 시점에 Cost Center로 비용이 발생하고, 송장 금액 차이는 회계 전표를 통해 CO 비용 또는 가격차이로 이어집니다.

S/4HANAACDOCA 중심 통합

ECC에서 나뉘어 있던 FI/CO 합계 테이블과 개별 항목 흐름을 S/4HANA에서는 ACDOCA 중심으로 이해했습니다. RACCT, RBUKRS, PRCTR, KOSTL, BTTYPE 같은 필드가 계정, 회사코드, 손익센터, 코스트센터, 트랜잭션 출처를 연결합니다.

Mentoring Activity

멘토링 자료는 SAP 개발 레이어를 빠르게 연결하는 데 초점을 맞춰 정리했습니다. DB 접근, OData 서비스, Fiori 화면, RAP 앱 구현이 어떤 순서로 맞물리는지 주차별로 정리했습니다.

Week 1 ADBC · AMDP · RAP/OData

Open SQL과 Native SQL 접근 차이를 비교하고, ADBC 직접 DB 접근과 AMDP의 Code-to-Data 처리 방식을 정리했습니다.

Week 2 Fiori Master-Detail

Binding Context, OData Key Path, bindElement, Association, $expand를 중심으로 리스트와 상세 화면 흐름을 정리했습니다.

Week 3 Gateway · OData Request Flow

Fiori 요청이 OData URI를 통해 Gateway로 전달되고, DPC_EXT CRUD 메서드와 HANA 조회로 이어지는 흐름을 정리했습니다.

Week 3+ Metadata Extension · BDEF · EML

UI Annotation, Behavior Definition, Validation, Determination, Action, EML 기반 READ/MODIFY ENTITIES 흐름을 정리했습니다.

Connected Project

SAP CODE에서 학습한 개발 스택과 CO 프로세스는 EverNiture ERP 구축 프로젝트로 이어졌습니다. 프로젝트에서는 PM으로 일정과 산출물 흐름을 관리하고, CO 담당으로 표준원가, 실제 활동단가, SKF, 배부, 수익성 분석 화면을 설계했습니다.
