Graduation Project · 2024.03 ~ 2024.12

Zkare Medical Record Management System

블록체인, IPFS, 하이브리드 암호화를 결합한 의료기록 관리 시스템입니다. 환자 중심 접근 권한, 기록 무결성 검증, 키 복구, 응급 접근 흐름을 1년간 설계하고 구현했습니다.

Project Overview

병원, 보험사, 연구기관으로 흩어지는 의료 데이터에서 환자의 통제권과 기록 무결성을 확보하는 것을 목표로 했습니다. 실제 의료 데이터는 암호화해 IPFS에 저장하고, 블록체인에는 CID, SHA-256 해시, 암호화된 키, 타임스탬프 등 검증에 필요한 메타데이터만 남기는 구조로 설계했습니다.

환자 직접 관리 선택적 접근 제어 무결성 검증 키 복구

Problem Definition

  • 의료 데이터가 병원별로 분산되어 환자가 자신의 기록을 통합적으로 관리하기 어려움
  • 보험사, 연구기관 등 제3자 공유 시 필요한 항목보다 더 많은 민감 정보가 노출될 수 있음
  • 중앙 저장소 기반 시스템은 해킹, 랜섬웨어, 내부 권한 오남용에 취약함
  • 개인키를 잃어버리면 암호화된 의료기록 접근권 자체가 사라지는 문제가 있음

Core Design

  • AES-256-GCM으로 의료기록 본문을 암호화하고, RSA-OAEP 2048-bit로 AES 키를 사용자별로 래핑
  • 환자, 의사, Master Account 역할을 분리해 등록, 조회, 응급 접근 권한을 제어
  • IPFS 변조 여부를 SHA-256 해시로 검증하고, 블록체인에 접근 로그와 메타데이터를 기록
  • 개인키 분실 상황에 대비해 Shamir's Secret Sharing 기반 2-of-3 보호자 키 복구 흐름 설계

Requirements

Functional

  • 사용자 키 생성과 공개키 등록
  • 환자 기본 정보 등록과 의료기록 생성
  • 역할 기반 접근 제어와 기록 조회
  • 보호자 승인 기반 키 복구 요청
  • 응급 상황을 위한 Master Account 접근

Non-functional

  • AES-256-GCM + RSA-OAEP 2048-bit 하이브리드 암호화
  • IPFS 저장으로 온체인 비용 절감
  • 블록체인 기반 접근 이력 투명성
  • React 기반 직관적인 사용자 흐름
  • 해시 검증을 통한 데이터 변조 탐지

System Architecture

화면은 React로 구성하고, 브라우저의 Web Crypto API에서 암복호화를 수행했습니다. Ethereum 스마트컨트랙트는 공개키, 역할, CID, 해시, 접근 권한을 관리하며, IPFS와 Pinata는 암호화된 의료기록 본문을 보관합니다.

Data Flow

Record Storage

  1. 의료기록 입력
  2. AES-256-GCM으로 기록 본문 암호화
  3. 의사, 환자, Master 공개키로 AES 키 각각 암호화
  4. 암호문을 IPFS에 업로드하고 CID 발급
  5. CID, 해시, 암호화 키, 타임스탬프를 블록체인에 저장

Record Retrieval

  1. 블록체인에서 CID와 해시 조회
  2. IPFS에서 암호화된 의료기록 다운로드
  3. SHA-256 해시로 변조 여부 확인
  4. RSA 개인키로 AES 키 복호화
  5. AES 키로 의료기록 본문 복호화

Smart Contracts

KeyRegistry 사용자 공개키와 Doctor, Patient, Master 역할을 관리합니다.
EncryptedMedicalRecord 환자 기본 정보, 의료기록 CID, SHA-256 해시, 사용자별 암호화 AES 키를 저장합니다.
KeyRecovery 3명의 보호자 중 2명 이상 승인 시 복구가 가능한 2-of-3 키 복구 요청을 관리합니다.

Roles & Access

Patient본인 의료기록 조회, 복호화, 키 복구 요청
Doctor환자 등록, 의료기록 생성, 진료 데이터 업로드
Guardian키 복구 요청 승인 또는 거절
Master Account응급 상황에서 제한적 접근, 접근 로그 기록

Key Recovery & Emergency Access

초기 설계에서는 개인키 분실 시 의료기록 접근이 영구적으로 불가능해지는 문제가 있었습니다. 이를 해결하기 위해 보호자 3명 중 2명 이상이 승인하면 복구가 가능한 구조를 설계했고, 복구 요청에는 24시간 유효기간을 두었습니다. 응급 접근은 Master 전용 암호화 키와 온체인 접근 로그를 함께 사용해 남용 가능성을 줄이는 방향으로 구성했습니다.

Testing & Results

  • KeyRegistry, EncryptedMedicalRecord, KeyRecovery 스마트컨트랙트 단위 테스트 통과
  • RSA 키 생성, 의료기록 암복호화, IPFS 업로드와 다운로드, 블록체인 트랜잭션 처리 검증
  • 의사 환자 등록, 기록 생성, 환자 조회, 2-of-3 키 복구, Master 응급 접근 시나리오 통합 테스트
  • 권한 없는 접근 차단, IPFS 데이터 변조 탐지, 잘못된 개인키 복호화 실패, Master 접근 로그 검증
